# Logging Standards（日志规约）

日志规约是保障系统可观测性和故障排查能力的重要规范，涵盖日志输出规范、日志级别选择、日志内容规范三个维度。

## 日志输出规范

### 日志框架选择

| 推荐 | 说明 |
|------|------|
| **SLF4J** | 日志门面，桥接各种日志实现 |
| **Log4j / Logback** | 常用日志实现 |

### 日志实现禁止混用

**强制**：在日志输出时，禁止 `System.out` / `System.err` 混用，必须使用日志框架。

**原因**：
- 日志框架可控制输出级别、目标媒介（控制台/文件）
- 统一日志便于集中分析、监控

## 日志级别规范

| 级别 | 使用场景 |
|------|---------|
| **ERROR** | 影响程序运行的异常，需要即时关注 |
| **WARN** | 存在潜在问题的警告信息 |
| **INFO** | 重要业务节点日志，记录系统运行状态 |
| **DEBUG** | 调试信息，仅本地开发使用 |
| **TRACE** | 最详细的跟踪信息 |

### 核心原则

1. **ERROR 级别**：记录异常信息后应升级处理，不应只记录日志
2. **DEBUG/TRACE 级别**：生产环境必须关闭，避免性能损耗
3. **INFO 级别**：日常运营关键节点日志

## 日志内容规范

### 应该记录

| 内容 | 说明 |
|------|------|
| 入口/出口参数 | 方法入口、出口参数 |
| 业务关键节点 | 订单创建、支付成功等 |
| 异常堆栈 | 完整的异常信息 |
| 性能数据 | 关键操作的耗时 |

### 禁止记录

| 禁止 | 原因 |
|------|------|
| **敏感信息** | 密码、密钥、个人身份信息 |
| **完整异常堆栈** | 在多次调用中使用 ERROR 级别重复记录 |
| **DEBUG 级别日志** | 敏感信息即使在 DEBUG 也不应输出 |

## 日志文件管理

| 规范 | 说明 |
|------|------|
| 应用日志与业务日志分离 | 便于问题定位 |
| 定期归档 | 避免日志文件过大 |
| 日志保留策略 | 根据合规要求和存储成本确定 |
