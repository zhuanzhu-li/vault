# Design Patterns in Architecture（架构设计中的设计模式）

设计模式在大型系统架构中按层级可分为四层：基础设施层、业务逻辑层、服务间通信层和架构层面。合理运用设计模式能显著提升系统的可维护性、可扩展性和可测试性。

## 一、基础设施层 — 创建型模式

| 模式 | 场景 | Java 示例 |
|------|------|----------|
| **工厂方法** | 对象创建逻辑复杂，需统一入口 | `ConnectionFactory`、`BeanFactory` |
| **抽象工厂** | 创建一系列相关对象的族 | 跨数据库的 DAO 族、不同主题的 UI 组件 |
| **建造者** | 构建不可变对象，链式调用 | `StringBuilder`、`Lombok @Builder` |
| **单例** | 全局唯一资源 | 配置中心、线程池、缓存实例 |

## 二、业务逻辑层 — 结构型 + 行为型模式

| 模式 | 场景 | Java 示例 |
|------|------|----------|
| **策略模式** | 多算法切换，运行时选择 | 支付方式（支付宝/微信）、促销计算 |
| **模板方法** | 算法骨架固定，具体步骤可变 | 业务流程审批、数据处理管道 |
| **责任链模式** | 多节点依次处理 | 过滤器链、拦截器链、日志链路 |
| **状态模式** | 状态机状态流转 | 订单状态（待支付→已支付→已发货→已完成） |
| **命令模式** | 请求封装为对象，支持撤销 | UI 命令队列、事务管理 |

## 三、服务间通信层 — 结构型模式

| 模式 | 场景 | Java 示例 |
|------|------|----------|
| **代理模式** | 远程调用、延迟加载、安全增强 | RPC 代理（Dubbo）、MyBatis 懒加载 |
| **装饰器模式** | 动态添加行为，不修改原类 | `BufferedInputStream`、AOP 切面 |
| **适配器模式** | 接口转换，兼容旧系统 | 多数据源适配、异构 API 集成 |
| **门面模式** | 统一入口，简化调用 | Service Facade、BFF（Backend for Frontend） |

## 四、架构层面 — 高级模式

| 模式 | 场景 | 核心价值 |
|------|------|---------|
| **CQRS** | 命令查询分离，读写分离 | 性能优化、模型简化 |
| **事件溯源** | 记录所有状态变更 | 审计日志、可追溯性 |
| **Saga** | 分布式事务协调 | 最终一致性 |
| **防腐层** | 隔离外部系统 | 保护领域模型、渐进式迁移 |
| **整洁架构** | 层级清晰，依赖单向 | 可测试性、可维护性 |

## 五、依赖方向控制

### 分层架构中的单向依赖原则

```
外层 → 内层（单向依赖）
─────────────────────
Controller → Service → Repository → Database
    ↓          ↓          ↓
   DTO      Domain    Mapper/Entity
```

| 原则 | 说明 |
|------|------|
| **依赖倒置** | 内层定义接口，外层实现 |
| **稳定依赖** | 稳定层（Domain）不应依赖不稳定层（Infrastructure） |
| **稳定抽象** | 稳定层应该是抽象的（接口），不稳定的实现细节应该依赖它 |

### 依赖注入容器的作用

| 功能 | 说明 |
|------|------|
| 实例管理 | 控制对象的生命周期（Singleton、Prototype） |
| 依赖解耦 | 调用方不直接依赖实现类 |
| 配置集中 | 外部化配置，便于切换实现 |
| AOP 支持 | 声明式事务、安全检查 |

## 六、架构反模式与修复

| 反模式 | 问题 | 修复方案 |
|--------|------|---------|
| **上帝类** | 一个类承担过多职责 | 按 SRP 拆分 |
| **循环依赖** | A→B→C→A | 依赖倒置、引入中间接口 |
| **早期抽象** | 过度设计不成熟的抽象 | YAGNI 原则，等待需求稳定 |
| **继承滥用** | 类层次过深，耦合严重 | 优先使用组合 |
| **重复 Switch** | 多处相同条件判断 | 策略模式 + 工厂模式 |
| **烛台式调用** | 调用层级过深 | 门面模式简化 |
