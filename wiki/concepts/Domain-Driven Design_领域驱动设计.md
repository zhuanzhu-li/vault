# Domain-Driven Design（领域驱动设计）

DDD（Domain-Driven Design，领域驱动设计）是一种通过深入理解业务领域来指导软件设计的方法论，由 Eric Evans 在《Domain-Driven Design: Tackling Complexity in the Heart of Software》一书中提出。

## 核心概念

### 限界上下文 (Bounded Context)

| 概念 | 说明 |
|------|------|
| 定义 | 一个边界内包含一个明确的领域模型 |
| 边界 | 明确哪些概念属于这个上下文，哪些不属于 |
| 通信 | 通过领域事件跨上下文通信 |

### 通用语言 (Ubiquitous Language)

| 要求 | 说明 |
|------|------|
| 团队共享 | 开发团队、业务方、领域专家共同定义 |
| 持续使用 | 在代码、文档、对话中贯穿一致 |
| 精确含义 | 每个术语在上下文中含义明确 |

## 战术设计 — 领域建模构件

### 实体 (Entity) vs 值对象 (Value Object)

| 特征 | 实体 | 值对象 |
|------|------|-------|
| 标识 | 有唯一标识 | 无唯一标识 |
| 可变性 | 可变状态 | 不可变 |
| 相等性 | 基于标识 | 基于属性值 |
| 生命周期 | 有生命周期管理 | 用完即弃 |

### 聚合根 (Aggregate Root)

| 职责 | 说明 |
|------|------|
| 保护不变性 | 确保整个聚合的业务规则一致 |
| 入口控制 | 所有对聚合内对象的访问通过聚合根 |
| 事务边界 | 一个聚合对应一个事务 |

### 领域服务 (Domain Service)

| 场景 | 说明 |
|------|------|
| 跨聚合操作 | 不属于任何聚合的业务逻辑 |
| 领域计算 | 复杂的业务规则计算 |
| 防腐协调 | 与外部系统交互的协调 |

### 领域事件 (Domain Event)

| 特点 | 说明 |
|------|------|
| 记录变更 | 表示领域已发生的事实 |
| 解耦 | 发布-订阅模式，跨限界上下文通信 |
| 可追溯 | 支持事件溯源 |

### 仓库 (Repository)

| 职责 | 说明 |
|------|------|
| 封装持久化 | 隐藏数据访问细节 |
| 接口在领域层 | 定义领域接口，基础设施实现 |
| 集合抽象 | 模拟对象集合的导航 |

## 分层架构与 DDD

### 经典分层

```
┌─────────────────────────────────────┐
│       用户界面层 (UI Layer)           │
├─────────────────────────────────────┤
│       应用层 (Application Layer)       │
│       └── 应用服务、用例编排           │
├─────────────────────────────────────┤
│       领域层 (Domain Layer)           │
│       ├── 聚合根、实体、值对象         │
│       ├── 领域服务、领域事件           │
│       └── 仓库接口、领域工厂           │
├─────────────────────────────────────┤
│       基础设施层 (Infrastructure Layer) │
│       ├── 仓库实现                     │
│       ├── 外部服务适配器               │
│       └── 技术服务（日志、消息等）      │
└─────────────────────────────────────┘
```

### 依赖方向

| 原则 | 说明 |
|------|------|
| **单向依赖** | 外层依赖内层，内层不依赖外层 |
| **依赖倒置** | 领域层定义接口，基础设施层实现 |
| **稳定依赖** | 领域层是稳定的，不依赖易变的外部 |

## DDD 与设计模式

| DDD 构件 | 对应设计模式 |
|---------|-------------|
| 聚合根 | 门面模式（对外统一入口） |
| 领域服务 | 策略模式、命令模式 |
| 工厂 | 工厂方法、抽象工厂 |
| 仓库 | 代理模式、仓储模式 |
| 领域事件 | 发布-订阅、观察者模式 |
| 防腐层 | 适配器模式 |

## DDD 与微服务

| 对应关系 | 说明 |
|---------|------|
| 限界上下文 | 微服务的边界 |
| 聚合 | 微服务内的业务能力 |
| 领域事件 | 微服务间通信 |
| 防腐层 | 防止外部系统污染领域模型 |

## 实施建议

| 阶段 | 重点 |
|------|------|
| **探索阶段** | 识别限界上下文，建立通用语言 |
| **深化阶段** | 完善领域模型，实施战术设计 |
| **落地阶段** | 渐进式重构，建立防腐层 |
